# Task Delegation Protocol

Defines the message schema, agent types, and transport layer for delegating tasks from the Telegram bot to the Dispatcher agent layer.

---

## Overview

```
Telegram bot
     │
     │  TaskPacket (via dispatchTask())
     ▼
Dispatcher (dispatcher/index.js)
     │
     │  Claude API call with agent system prompt
     │  + CHRONICLE context injection
     ▼
Result (ResultPacket)
     │
     ▼
Telegram bot → sends result to user
```

The Dispatcher is an **in-process module** — `dispatchTask(packet)` returns a Promise. No external queue or HTTP server needed at this stage. The bot `await`s it while keeping the typing indicator active.

---

## Task Packet

Sent by the bot to `dispatchTask()`:

```json
{
  "taskId": "uuid-v4",
  "type": "research | code | write | analyze",
  "goal": "One-sentence objective in plain English",
  "context": {
    "chronicle_refs": ["session-id-prefix"],
    "inline": "Optional extra context text (pasted by user, etc.)"
  },
  "constraints": {
    "model": "claude-sonnet-4-6",
    "max_tokens": 4096
  },
  "deliverable": "markdown | json | text",
  "callback": {
    "type": "telegram",
    "chat_id": 123456789
  }
}
```

### Field notes

| Field | Required | Notes |
|-------|----------|-------|
| `taskId` | yes | UUID generated by the bot at dispatch time |
| `type` | yes | Selects system prompt and tone |
| `goal` | yes | The user's intent verbatim (1 sentence) |
| `context.chronicle_refs` | no | Session IDs to pull from CHRONICLE as context |
| `context.inline` | no | Any additional context the user pasted |
| `constraints.model` | no | Defaults to `claude-sonnet-4-6` |
| `constraints.max_tokens` | no | Defaults to `4096` |
| `deliverable` | no | Defaults to `markdown` |
| `callback` | yes | Where to send the result |

---

## Result Packet

Returned by `dispatchTask()` when the Promise resolves:

```json
{
  "taskId": "uuid-v4",
  "status": "completed | failed | timeout",
  "result": "The agent's response text",
  "model": "claude-sonnet-4-6",
  "usage": {
    "input_tokens": 1234,
    "output_tokens": 567
  },
  "elapsed_ms": 8432
}
```

On `failed` or `timeout`, `result` contains the error description for display to the user.

## Agent Types

Each `type` maps to a focused system prompt. All agents receive:
1. The system prompt for their type
2. Any CHRONICLE context (pulled via `chronicle_refs` or `ctx-search` auto-lookup)
3. The user's `goal` as the user message

| type | System prompt focus | Default model |
|------|--------------------|-|
| `research` | Synthesize and summarize — cite sources, structure findings | claude-sonnet-4-6 |
| `code` | Implement cleanly — no fluff, production-ready, explain key decisions | claude-sonnet-4-6 |
| `write` | Professional prose — clear, concise, appropriate tone for audience | claude-sonnet-4-6 |
| `analyze` | Structured analysis — identify patterns, risks, trade-offs, recommendations | claude-sonnet-4-6 |

## Bot Command

```
/dispatch <type> <goal>
```

Examples:
```
/dispatch research What are the trade-offs of LanceDB vs Meilisearch for semantic search?
/dispatch code Write a Node.js module that chunks text into 512-token windows
/dispatch analyze Review the current LiteLLM gateway config for cost optimization opportunities
/dispatch write Draft a README for the dispatcher module
```

### Bot flow

1. User sends `/dispatch <type> <goal>`
2. Bot validates `type` (must be one of: research, code, write, analyze)
3. Bot generates `taskId` (UUID)
4. Bot sends typing action
5. Bot calls `await dispatchTask(packet)` (blocking, with typing keepalive loop)
6. Bot sends result to chat; on failure sends error message

### Typing keepalive

Telegram's typing indicator expires after ~5 seconds. For long tasks, the bot re-sends `sendChatAction('typing')` every 4 seconds while awaiting the result:

```js
const keepalive = setInterval(() => bot.sendChatAction(chatId, 'typing'), 4000)
const result = await dispatchTask(packet)
clearInterval(keepalive)
```

## Context injection

Before calling the Claude API, the Dispatcher:

1. If `chronicle_refs` is provided — loads those sessions from CHRONICLE
2. Runs `getContextForPrompt(goal)` (existing ctx-search integration) to pull relevant past sessions
3. Prepends both as a `system` message block before the user's `goal`

This reuses the same context injection already in `handleAsk()` in `bot.js`.

## Future extensions

| Feature | Notes |
|---------|-------|
| Async dispatch | Fire-and-forget: bot sends "Task dispatched, I'll message you when done" then notifies on completion |
| Task status polling | `/task <taskId>` command to check status of long-running dispatches |
| BATON handoff | Result packet extended with `baton_ref` pointing to next agent in chain |
| Multi-turn agents | Dispatcher runs agent loops with tool use (requires Claude API tool_use) |
